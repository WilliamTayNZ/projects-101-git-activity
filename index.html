Activity 1 — Change & Pull Request

We will use the core workflow: branch → change → commit → push → PR → merge.

  1. Suppose you wanted to remove this red text "remove me".
  2. Create a new branch: git checkout -b chore/remove-red-text
  3. Delete the red text and save the file.
  4. Commit & push: 
    git add .
git commit -m "chore: remove red text"
git push -u origin chore/remove-red-text
  5. On GitHub, open a Pull Request to main, add a clear title/summary, then merge.
Why we do this
  • Creating branches for specific changes keeps work focused and isolated.
  • Clear commit and PR messages help you and your collaborators.
  • PRs allow collaborators to review code and resolve potential conflicts, ensuring a smooth merge into main.

Activity 2 — Resolve a Merge Conflict

Suppose while working on separate branches, you and a teammate both changed the same line in a file. When Git tries to merge, it doesn’t know whose edit to keep — that’s a merge conflict.

  1. Create a new branch: git checkout -b chore/change-heading

    Tip: You can run git branch to see which branches are available on your local repository.

  2. Edit the main heading in index.html, for example: <h1>Git Workshop Activity</h1> and commit your change.
  3. Switch back to main: 
    git checkout main
    Change the same heading line differently (e.g. <h1>Git & GitHub Hands-on</h1>) and commit.
  4. Now try to merge your branch into main: 
    git merge chore/change-heading
    Git will stop and show a merge conflict.
  5. Open index.html to view the conflict markers (<<<<<<<, =======, >>>>>>>). Edit the file to the final version you want, then: 
    git add index.html
git commit
    to finish the merge.
Why this happens

Both branches edited the same line differently. Git cannot decide which version to keep, so it asks you to resolve the conflict manually.
